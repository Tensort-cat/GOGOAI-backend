# 测试问题（按能力分层）

## 1️⃣ 基础检索（验证 chunk 命中）

这些问题应该能直接命中文档中的单一段落：

* Yuko Chat 使用了哪些技术栈？
* 系统中 Redis 的作用是什么？
* Kafka 在系统中扮演什么角色？
* 用户登录后返回的是什么？
* 系统支持哪些会话类型？

---

## 2️⃣ 跨段落整合（验证上下文拼接）

需要从多个 chunk 拼接信息：

* 描述 Yuko Chat 的完整消息流程
* 系统是如何保证性能的？
* 系统的安全措施有哪些？
* 文件存储是如何实现的？

---

## 3️⃣ 多跳推理（验证 reasoning + retrieval）

需要结合多个模块理解：

* 为什么系统需要同时使用 Kafka 和 Redis？
* 如果 Kafka 挂了，会影响哪些功能？
* 为什么要使用 JWT 而不是 session？
* 系统如何实现高并发下的消息处理？

---

## 4️⃣ 反向验证（防 hallucination）

这些问题文档**没有答案**，用来检测幻觉：

* 系统是否支持视频通话？
* 是否使用了 Elasticsearch？
* 用户密码是否使用 MD5 加密？
* 是否支持多端同步（明确写没写）

---

## 5️⃣ 精细定位（验证 chunk 精度）

用于测试 embedding + chunk 切分是否合理：

* “数据库分表”是按什么维度进行的？
* WebSocket 在系统中出现在哪个流程步骤？
* 网关层的主要职责有哪些？

---

# 三、进阶测试建议

如果你想把 RAG 测得更“狠”一点，可以进一步：

### ✔ 控制变量测试

* 改 chunk size（128 / 256 / 512）
* 改 top_k（3 / 5 / 10）

### ✔ 加噪声文档

插入一段无关内容，比如：

```md
## 附录：猫的分类
猫分为家猫和野猫...
```

看模型会不会误检索。

### ✔ 重复信息测试

写两段类似但不完全一样的内容，测试排序能力。

---

# 四、你可以这样验证效果

重点看这几个指标：

* 是否命中正确 chunk
* 是否引用了错误上下文
* 是否出现 hallucination
* 多段信息能否正确拼接
